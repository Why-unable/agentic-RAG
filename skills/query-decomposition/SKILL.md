# 问题拆解 (Query Decomposition) Skill

本技能提供了处理复杂用户查询（包含多个维度、实体或逻辑依赖关系）的强制性指导原则。直接检索这类复杂查询通常会导致失败或返回大量无效噪音。

## 何时使用本技能 (When to Use)

当用户的查询符合以下任何一种情况时，必须使用本技能：
- **对比与聚合 (Comparison or Aggregation)**：要求对比两个或多个实体，或对多个主题进行总结（例如：“Llama 3 和 GPT-4 的区别”、“总结苹果、谷歌和微软的财报”）。
- **多跳依赖 (Multi-hop / Logical Dependency)**：问题的某部分答案依赖于另一部分的答案（例如：“负责处理支付服务宕机异常的那个团队主管是谁？”）。
- **复杂过滤 (Complex Filtering)**：查询中混合了明确的元数据限制与语义意图（例如：“找一下今年第四季度关于差旅的邮件”）。

## 强制性拆解模式 (Required Breakdown Modes)

在执行任何外部搜索或 RAG 检索工具之前，你**必须**分析查询并应用以下核心模式之一（或组合）：

### 1. 并行拆解 (Parallel Decomposition)
**适用场景**：对比类、多实体类问题。
**执行步骤**：
1. 识别查询中所有独立的实体/主题。
2. 为每个实体构建一个独立的、完整的原子查询。
3. 执行所有原子查询以检索各自的答案。
4. **输出格式**：将检索到的结果聚合成结构化的对比或总结报告。

### 2. 串行拆解 (Sequential Decomposition)
**适用场景**：多跳推理或属性依赖类问题。
**执行步骤**：
1. 识别回答主问题所需的隐藏中间变量。
2. **步骤 1**：执行具有针对性的原子查询，找到这个中间变量。
3. **步骤 2**：将检索到的变量值注入到下一个原子查询中，以找出最终答案。
4. 任何一步如果失败必须停止，并报告依赖链中的信息缺失。

### 3. 显式条件拆解 (Filter-Based Decomposition)
**适用场景**：包含显式元数据过滤条件的查询。

## 输出格式与任务规划 (Output Format & Task Planning)

当系统提供了任务规划工具（如 `write_todos`）时，请**避免在对话正文中重复打印大段的 Markdown 拆解日志**。一旦你完成了多维查询的拆解分析，请**直接使用 `write_todos` 工具**将你拆解出的原子查询（q1, q2...）记入任务列表（Todo List）。

**使用规范**：
1. 在任务列表的 `content` 字段中，清晰写出每一个拆解后的“原子查询”意图以及执行方式（如：“查询Llama 3的上下文窗口大小”、“查询GPT-4的上下文窗口大小”）。
2. 将当前需要立即执行的步骤标记为 `in_progress`。
3. 逐个执行你的拆解步骤：完成一个原子查询就将其状态更新为 `completed`，再继续执行下一个。

**严禁**跳过针对多维查询的拆解步骤。务必在将问题传给知识检索工具前，先把原子化重构的步骤录入待办事项列表。
